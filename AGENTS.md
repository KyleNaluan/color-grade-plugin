# Project agent memory

This file is the project's committed home for project-intrinsic agent knowledge: build, test, release, architecture, and sharp-edge notes that should travel with the code.

- Domain vocabulary lives in `CONTEXT.md`; use those terms verbatim. Product context: PRD in issue #1 and `docs/prd.md`; ADRs in `docs/adr/`.
- Layout: `src/core` (pure color math), `src/themes` (data-only looks), `src/panel` (Preact UI), `src/host` (typed bridge boundary + CEP impl), `src/extendscript` (ES3 `host.jsx`, AE DOM ops only, zero color math).
- Pixel access goes through the `FrameSource` interface (`src/host/frameSource.ts`, ADR 0001): comp time in, normalized Float32 pixels out - consumers never learn the acquisition method. v1 backend is render-to-file (`renderFrameSource.ts` + `cepFrameFileIO.ts` + ExtendScript `CG_renderFrame`, 16-bit TIFF primary / PNG fallback, decoders in `frameFile.ts`). Tests use the file-backed `fileFrameSource.ts` fed with fixtures, so pixel consumers run without a host.
- Commands: `npm test` (Vitest), `npm run lint` (core-purity check), `npm run typecheck`, `npm run build` (Vite + `scripts/assemble-cep.mjs` -> installable CEP extension in `dist/`; install steps in `docs/cep-install.md`). Theme-tuning loop: `npm run spike -- <frame.tif> <theme>` (see `scripts/spike.ts` header). The spike loop also prints a "grade impact" report (`src/core/analysis/gradeImpact.ts`): skin-region hue/chroma shift and overall cast magnitude/direction, applying the built transform to real pixels - use this as the evidence source for theme tuning decisions, not eyeballing stats alone. CI (`.github/workflows/ci.yml`) runs `npm test`, `npm run lint`, and `npm run typecheck` on Node 22 for pushes and PRs to `main`.
- Themes are data only (target stats + authored overrides + knobs); all look logic lives in `src/core/engine/engine.ts`. Strength 0 must stay exact identity and output bounded in [0,1] - tests enforce both. All 3 shipping themes (`src/themes/{teal-orange,warm-film,cool-noir}.ts`) were tuned against the 12 fixture families via the spike loop (issue #6); each theme file's header comment records the before/after evidence and which footage families it suits. Two cross-cutting findings from that pass: `vlog-indoor-flat-tricky-skin` is an uncorrected-tungsten-WB outlier that drives outsized cast readings on all 3 themes (a fixture/Correct-stage gap, not a theme problem); and `rec709-outdoor` / `vlog-outdoor-bright` show a real ~10-14deg skin hue shift under all 3 themes, from partial-weight pixels at the soft skin-wedge's edge (`skinWedgeWeight` in `src/core/analysis/stats.ts`) getting only partial protection by design - a candidate for the deferred corrective-pinning improvement (see `CONTEXT.md`), not something theme-data tuning alone can close.
- `src/core` purity is enforced by `scripts/check-core-purity.ts` (custom checker, not eslint: typescript-eslint's peer range rejects this repo's TypeScript 7). Core may only import relative modules inside `src/core`.

- Correct stack (issue #5): `src/core/color/decode.ts` (`decodePixelToRec709`) is the single per-pixel decode-to-Rec.709 composition, shared by `analysis/stats.ts` (bulk footage decode) and `lut/decodeLut.ts` (`bakeDecodeLut`, which bakes it into a Decode LUT). Panel orchestration is `panel/correctStack.ts`; the bridge command is `Bridge.setCorrectProfile`, implemented in ExtendScript as `CG_setCorrectProfile` (matchNames `ADBE Apply Color LUT2`, `ADBE Lumetri`). Decode LUTs default to a 65-point grid, not the theme-grade 33-point default: a log profile's highlight rolloff is steep enough that 33 points under-resolves it (see `bakeDecodeLut`'s doc comment and `tests/unit/decodeLut.test.ts`). The Correct tab's UI is a single footage-profile `<select>` (`data-testid="footage-profile"`) that is the one source of truth for the clip's log format: it drives BOTH the analyze decode and, via an explicit **Apply correction** button (`apply-correct`), the Correct stack. `panel/correctStack.ts`'s `applyCorrectForProfile(bridge, profileKey, layerId)` resolves the key to `PROFILES[key]` and decides log-vs-standard via `isLogProfile` (`src/core/color/index.ts`, standard = `STANDARD_PROFILE_KEY='rec709'`) - no hardcoded profile in the apply path; a log key bakes+applies that profile's Decode LUT, `rec709` removes it. The selector resets to `rec709` per-clip and is guarded by `applyGeneration`/`targetKey` against stale applies. Extending to more camera profiles (`multi-camera-profiles`) is just adding entries to `PROFILES`.
- `cep.fs` paths must be plain native OS paths, but `__adobe_cep__.getSystemPath('userData')` returns a `file://` URI on Windows (e.g. `file:///C:/Users/.../AppData/Roaming`). Feeding that URI straight to `cep.fs.writeFile` is what caused the Windows-AE `cep.fs.writeFile failed (err 1)` on the Correct Decode LUT - NOT payload size or a missing encoding argument (a plain path writes a 7MB .cube with no encoding arg just fine; confirmed in the AE CEP console). `cepBridge.ts` `stageTempFile` normalizes via the pure, unit-tested `systemPathToNative` (`src/host/cepPath.ts`, tested in `tests/unit/cepPath.test.ts`): strips the `file://` scheme, `decodeURIComponent`s (`%20`), and drops the leading `/` before a Windows drive letter. Apply this helper to any `getSystemPath`-derived path before it reaches `cep.fs`. Paths from ExtendScript `Folder.temp.fsName` (the render read-back in `cepFrameFileIO.ts`) are already native, so they need no normalization. This boundary is untestable here - the captain verifies in AE via the Correct smoke-checklist item in `docs/cep-install.md`.
- Frame render range (issue: "render produced no file" on timecode-offset comps): `CG_renderFrameToTiff` in `host.jsx` renders one frame via the render queue, and `RenderQueueItem.timeSpanStart` lives in the comp's `displayStartTime` domain, whereas `time` (from `comp.time`) is comp-internal 0-based. On a comp with a large non-zero `displayStartTime`, setting `timeSpanStart = time` (0-based) falls outside the render range -> no frame. Fix: `timeSpanStart = comp.displayStartTime + time`. The PNG fallback's `comp.saveFrameToPng(time, ...)` takes comp-internal time and needs NO shift. Verified via the non-zero-Start-Timecode smoke-checklist item.
- Apply Color LUT's file-picker property is not officially documented as scriptable; both `CG_ensureDecodeLut` (Correct) and `CG_ensureGradeLut` (Grade) set it via `effect.property(1).setValue(file)` as the best-known approach - verify it actually resolves the LUT file each release via the smoke checklist.
- Grade stack (issue #7): the pure bake seam is `lut/gradeLut.ts` (`bakeGradeLut` = `bakeLut(buildTransform(stats, theme, opts))`, default 33-point grid). Panel orchestration is `panel/gradeStack.ts` (`applyThemeGrade`); the bridge command is `Bridge.applyGrade`, implemented as `CG_applyGrade`, which writes `grade_<layerId>.cube` + its recipe `grade_<layerId>.json` into `.colorgrade/` and puts one Managed `[cg]` adjustment layer at the top of the comp carrying an Apply Color LUT effect. Grade analysis is post-Correct by construction: the FrameSource renders the comp *with* the clip's Correct stack applied, so `applyThemeGrade` decodes through the identity `REC709` profile (no second software decode) - this is the hard pipeline-order requirement. On a re-grade, `applyThemeGrade` disables the existing `[cg]` grade layer (via `Bridge.setGradeLayerEnabled` / `CG_setGradeLayerEnabled`) around the analysis render and re-enables it in a `finally`, so the analyzed pixels stay post-Correct instead of feeding the prior grade back in. Knob recompute-from-cached-stats and per-clip grade recall are deferred (issues #11+); the tracer bakes with default knobs and persists the recipe inputs for that later work.
- Chroma-overshoot guard (`toneStretchChromaGuard` in `engine.ts`, folded into `buildTransform`): stat-matching a low-dynamic-range / degraded source toward a wide-range target used to neon-blow-up chroma - the tone-curve stretch relocates mid pixels into near-empty shadow/highlight bands where auto `bandScale` (<=2x) and the per-pixel `kA`/`kB` std-ratio gain (<=1.8x) compound (scout root cause: `firstmate/data/cg-agent-grade-s7/report.md` Exp A). The guard keys on `stretch = target luma range / source luma range (p5..p95)`: inert below `STRETCH_KNEE=1.5` (ordinary sources -> byte-identical output), ramping to a `gain` floor of 0.2 on the *auto* amplification by `stretch~3`, plus an auto LAB-chroma soft ceiling (3-6x target max band chroma) only when the theme authored no `chromaShape.softLimit`. In the engaged large-stretch regime the guard's `gain` bounds the *total* auto-driven chroma amplification by design - this includes an authored `chromaGain` (it folds into the same `scale` product), since this is exactly the blowout-prevention regime; the auto soft ceiling, though, still defers to an authored `chromaShape.softLimit`, and outside the guard's engaged regime hand-authored intent is untouched. Ported to the native C++ engine in Phase 2 (`toneStretchChromaGuard` in `native/ColorGradeFX/core/Engine.h`, folded into `buildTransform`).

The Footage param drives a native **decode stage** applied in EVERY LUT Source mode (pipeline order always decode-then-LUT, captain directive): Auto composes `grade(decodePixelToRec709(x))` into one LUT (continuous), Embedded/External resample their raw LUT through decode (`ComposeDecodeIntoLut`: `sampleLut(rawLut, decode(x))`); both keep the per-pixel apply a single trilinear sample so CPU/GPU stay identical (Rec.709 = no-op). INSTANCE KEY (`SeqKey`): the registry is keyed on a uid in **sequence data** (flat POD `CG_SequenceData`, reseeded on SEQUENCE_SETUP/RESETUP so duplicated/reloaded effects never share a key). This fixed a live-AE bug: `in_data->effect_ref` is NOT stable across command types (differs USER_CHANGED_PARAM vs SMART_PRE_RENDER), so effect_ref-as-key made publish/drain miss the window and the round-trip was dead. Bridge flow: effect->window `publishSnapshot` from BOTH `PreRender` (opportunistic, locked value copy) AND the idle hook, which POLLS the effect's param streams (`AEGP_GetNewStreamValue`) and publishes on change - the idle-hook poll is the reliable EC->window path because `PreRender`'s `sequence_data`/key is unreliable on render threads (this was the last EC->window-not-updating fail); monotonic `revision` + the window's `IsAnyItemActive` mid-drag guard prevent stomping an active drag. window->effect edits applied by the same **global AEGP idle hook** registered via PICA (`AEGP_RegisterSuite5::AEGP_RegisterIdleHook` after `AEGP_RegisterWithAEGP` - NO separate AEGP plugin/PiPL) that drains each open window and writes streams (`AEGP_StreamSuite5` GetNewEffectStreamByIndex+SetStreamValue) inside one `AEGP_StartUndoGroup`/`EndUndoGroup` per tick; it reaches the instance via a per-key `AEGP_EffectRefH` captured on button-open (`AEGP_PFInterfaceSuite1::AEGP_GetNewEffectForEffect`) and cheap-noops when nothing is pending. All AEGP wrapped in `catch(...)` (absent gracefully on non-AEGP hosts); requires the SDK's **`AEGP_SuiteHandler.cpp`** compiled in + a client-provided `AEGP_SuiteHandler::MissingSuiteError()` (throws). The in-`UserChangedParam` `CHANGED_VALUE` drain stays as a secondary flush. Lifecycle: `GLOBAL_SETDOWN`->`shutdownAll`, `SEQUENCE_SETDOWN`->close+dispose effect ref (no orphan windows). VCXPROJ: links `d3d11.lib;dxgi.lib` (imgui backends `#pragma comment(lib)` d3dcompiler/gdi32/dwmapi; XInput dynamic), compiles `AEGP_SuiteHandler.cpp`, adds `third_party/imgui[/backends]` includes; editor sources compile in ALL FOUR configs (build Debug/Release x CPU/`--gpu`); NOMINMAX/MultiByte/stdcpp17 already satisfy ImGui. LINK-VERIFY WITHOUT CLOSING AE: pass `-p:AE_PLUGIN_BUILD_DIR=C:\dev\cg-verify-out` (a WSL `export` does NOT cross to Windows MSBuild - must be an explicit `-p:` prop) to relink to a scratch dir instead of the locked MediaCore `.aex`. DEFERRED (captain-verified in AE, later phases): live preview (Phase 4, now landed - see next bullet) + scopes (Phase 5). AEGP DOM round-trip + undo are captain-verified in AE. Adding a param shifts the `CG_*` enum but all refs are by name; PiPL `AE_Effect_Version` unchanged, so no version-mismatch risk.
- Native Phase 4 - live clip preview (`native/ColorGradeFX/editor/PreviewCache.h` + the idle-hook driver in `ColorGrade.cpp` + the D3D texture path in `EditorWindow.cpp`): the editor window shows the actual clip frame, centered + letterboxed, refreshing on timeline scrub and any effect-param change. Design + AE-verification detail is in `native/docs/adr-editor-ui.md` (the "Phase 4" section). Key points: the frame is checked out DOWNSTREAM of this effect (`AEGP_LayerRenderOptionsSuite2::AEGP_NewFromDownstreamOfEffect` -> `AEGP_RenderSuite5::AEGP_RenderAndCheckoutLayerFrame`), so the pixels are already decode+graded - **the V-Log decode invariant holds by construction**, no second decode path. `AEGP_NewFrom*` seeds the options' Time to the layer's current time, read back via `AEGP_GetTime` as the scrub position (no comp-time lookup); `NewFromDownstreamOfEffect` is UI-thread-only, satisfied by the idle hook. The 8-bit ARGB receipt world is copied ARGB->RGBA (alpha forced 255 - premult opaque clip), decimated to <=960px long side, uploaded to a per-window `DXGI_FORMAT_R8G8B8A8_UNORM` texture on the window's UI thread, drawn with `ImDrawList::AddImage` + `letterboxFit`. Interactivity comes from a bounded per-instance LRU cache (16 frames) keyed by (frame time + grade-param fingerprint from the same stream poll that feeds the controls); the pure state machine `decidePreviewAction` picks UpToDate/ServeCached/Render. Checkout is SYNCHRONOUS on the idle thread (sanctioned because the cache makes repeats free); **every** checkout is checked back in unconditionally via a `ScopedCheckin` RAII guard (a leaked receipt destabilizes AE - load-bearing). PORTING/TESTING: the risky logic (cache keying/eviction, scrub/param decision, letterbox fit, once-and-always checkin) is pure in `PreviewCache.h`, headless-tested by `npm run native:preview-test` (`tests/editor/preview_test.cpp`, g++/clang, NOT in CI); the AEGP checkout + D3D upload are captain-verified in AE. All 4 configs (Debug/Release x CPU/`--gpu`) compile clean. DELETION LIFECYCLE (captain findings, rounds 1-2 - CRITICAL): deleting the effect while its window is open does NOT reliably reach `SEQUENCE_SETDOWN`, so the idle hook must self-detect it. NEVER touch ANY stream of a possibly-stale `AEGP_EffectRefH`: a stale ref makes `AEGP_GetNewStreamValue` raise modal 5027::247 (NO_DATA) AND `AEGP_GetNewEffectStreamByIndex`/`AEGP_GetStreamType`-by-index raise "invalid index in indexed group" then CRASH AE. Final design: the idle hook keeps NO AEGP handles at all (effect ref, layer handle, AND comp handle all go stale across delete/undo). At button-open it captures only STABLE IDS: the comp's project-item id (`AEGP_GetItemFromComp`+`AEGP_GetItemID`) + the layer-ID (`AEGP_GetLayerID`) + our installed-effect key. Each tick `ResolveLiveEffect` re-resolves everything from those ids (TRI-STATE, liveness = genuine PROJECT/COMP MEMBERSHIP not call-failure): (1) re-resolve the comp handle by walking the LIVE project item list (`AEGP_GetFirstProjItem`/`AEGP_GetNextProjItem`) for our item id (`ResolveCompByItemID`); (2) `AEGP_GetLayerFromLayerID` within that comp; (3) enumerate that layer's effects by installed key. Alive (use fresh ref for read/write/render, disposed at end of pass; reset counter) / ConfirmedGone (live comp+layer, no effect match => close promptly) / CannotVerify (comp or layer not a live member = comp/layer/source deleted, or partial context). WHY membership not call-failure (rounds 4-5): a deleted layer OR comp zombies in AE's undo buffer where a cached handle keeps resolving, so failed-enumeration counting never fired and the window lingered; re-resolving from the live project/comp tree by id fails on delete and re-succeeds on undo with the CURRENT handle - which also fixes the intermittent edit-after-undo modal by re-deriving the live effect for writes. A single CannotVerify never force-closes (transient guard); a PERSISTENT run closes at `CG_VERIFY_FAIL_LIMIT` (~0.8s). While not-Alive the hook DROPS the orphaned window's queued edits (writes only in the Alive branch). EXACTLY ONE WINDOW PER EFFECT: the registry keys on the sequence-data uid, but delete+undo can reseed it (SEQUENCE_RESETUP), so Open Editor first calls `CloseDuplicateWindowsForEffect` (closes any window with the same captured comp-item+layer id under a different key) before opening/adopting the current key. `ReadStreamOneD` also rejects `NO_DATA` defensively. Multi-instance limitation: two CG effects on one layer match the first (no per-instance id from the idle hook). Also: AE param names are narrow/CP-1252, so use ASCII (the Open Editor button label is "Open Editor..." not a UTF-8 ellipsis, which rendered as mojibake). DEFERRED: async checkout (`AEGP_RenderAndCheckoutLayerFrame_Async` - SDK prefers it for passive redraws; the state machine is already shaped for one in-flight request) if scrub interactivity needs it; the cache fingerprint keys on time + OUR params, so an unrelated edit (another effect / source swap without a time change) can leave the preview stale until the next scrub/param nudge; scopes + before/after are Phase 5.
- Native Phase 5 - in-effect analysis + live scopes + before/after (`native/ColorGradeFX/editor/Analysis.h` + `Scopes.h` + split geometry in `PreviewCache.h`, driven from the idle hook in `ColorGrade.cpp`, drawn in `EditorWindow.cpp`). Full design + the AE verification checklist (items 9-12) are in `native/docs/adr-editor-ui.md` ("Phase 5" section). Key facts: (1) ANALYSIS is in-effect and multi-frame - the idle hook samples several frames across the layer duration UPSTREAM of this effect (`AEGP_NewFromUpstreamOfEffect`, UI-thread-only like the Phase-4 downstream checkout), decodes each to Rec.709 via the footage profile (the SAME `decodePixelToRec709` the Auto bake composes, so stats and grade agree and V-Log is never analysed as raw log), and runs the ported `computeStats` over the union. It is INCREMENTAL: exactly one upstream checkout per idle tick (never on the editor's D3D thread), so AE never stalls; every checkout uses the Phase-4 `ScopedCheckin`. Measured stats go into a process-global `g_analyzedStats` (mutex; idle-hook writes, render reads) and are injected over `recipe.sourceStats` at render (`InjectAnalyzedStats`) ONLY while the footage profile still matches - so switching Rec.709<->V-Log never applies the other decode's stats. Completion calls `PF_TouchActiveItem` (PF `AdvItemSuite1`) to force the active comp to re-render. Persisting analysed stats to arb-data/project is DEFERRED with grade-recipe persistence (issues #11/#12) - analysis re-runs on open. (2) REBAKE/DEBOUNCE: re-analysis triggers only on an analysis-relevant change (footage profile or clip span - NOT the grade knobs, which only rebake the LUT), debounced N idle ticks (`AnalysisDebounce`) so a fast toggle doesn't thrash checkouts, with a small per-instance fingerprint->stats results cache (re-selecting a profile is instant), plus a forced-run "Analyze" button. (3) SCOPES are computed on the window UI thread from the SAME graded preview frame (`Scopes.h` synthesises waveform/histogram/vectorscope into small RGBA8 images), uploaded + drawn as GPU textures via `ImDrawList::AddImage` - same GPU path as the preview. (4) BEFORE/AFTER: toolbar toggles After/Before/Split; "Before" = the upstream frame DECODED to Rec.709 (never raw log), checked out on the idle hook only when `wantsBeforeFrame`, cached by (time+footage); split uses `splitViewGeometry`. All pure logic (schedule/job/debounce, scope binning, split geometry) is headless-tested by `npm run native:analysis-test` + `native:scopes-test` (g++/clang, NOT in CI); AEGP checkout + `PF_TouchActiveItem` + D3D uploads are captain-verified.

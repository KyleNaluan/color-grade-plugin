# Project agent memory

This file is the project's committed home for project-intrinsic agent knowledge: build, test, release, architecture, and sharp-edge notes that should travel with the code.

- Domain vocabulary lives in `CONTEXT.md`; use those terms verbatim. Product context: PRD in issue #1 and `docs/prd.md`; ADRs in `docs/adr/`.
- Layout: `src/core` (pure color math), `src/themes` (data-only looks), `src/panel` (Preact UI), `src/host` (typed bridge boundary + CEP impl), `src/extendscript` (ES3 `host.jsx`, AE DOM ops only, zero color math).
- Pixel access goes through the `FrameSource` interface (`src/host/frameSource.ts`, ADR 0001): comp time in, normalized Float32 pixels out - consumers never learn the acquisition method. v1 backend is render-to-file (`renderFrameSource.ts` + `cepFrameFileIO.ts` + ExtendScript `CG_renderFrame`, 16-bit TIFF primary / PNG fallback, decoders in `frameFile.ts`). Tests use the file-backed `fileFrameSource.ts` fed with fixtures, so pixel consumers run without a host.
- Commands: `npm test` (Vitest), `npm run lint` (core-purity check), `npm run typecheck`, `npm run build` (Vite + `scripts/assemble-cep.mjs` -> installable CEP extension in `dist/`; install steps in `docs/cep-install.md`). Theme-tuning loop: `npm run spike -- <frame.tif> <theme>` (see `scripts/spike.ts` header). The spike loop also prints a "grade impact" report (`src/core/analysis/gradeImpact.ts`): skin-region hue/chroma shift and overall cast magnitude/direction, applying the built transform to real pixels - use this as the evidence source for theme tuning decisions, not eyeballing stats alone. Two dev-only library tools (not core, not CI): `npm run gallery` (`scripts/genThemeGallery.ts`) renders a before/after PNG gallery of every registered theme against the fixture frames, and `npm run theme:health` (`scripts/themeLibraryReport.ts`) scans the whole `THEMES` registry printing per-theme cast/skin grade-impact - run either to spot outliers when adding or auditing themes. CI (`.github/workflows/ci.yml`) runs `npm test`, `npm run lint`, and `npm run typecheck` on Node 22 for pushes and PRs to `main`.
- Themes are data only (target stats + authored overrides + knobs); all look logic lives in `src/core/engine/engine.ts`. Strength 0 must stay exact identity and output bounded in [0,1] - tests enforce both. The library is 24 looks registered in `src/themes/index.ts`: the original 3 (`teal-orange,warm-film,cool-noir`, issue #6) tuned against the 12 fixture families via the spike loop, the manual-only `none-manual` base, and a 20-look curated expansion (`fm/cg-theme-library`) authored via the `npm run author` gradeGuard pipeline. Every theme file's header comment records its before/after grade-impact evidence and which footage families it suits (adding a look = a new data file in `src/themes/` + a registry entry, no engine/type change). Design notes worth carrying: `monochrome-bw`/`sepia` use `chromaGain:0` (+ `skinProtection:0`, `strength:1.0`) to force a true grayscale since the engine has no desaturate knob - their large skin/cast numbers are the look working as designed; `cinematic-green`/`neon-cyberpunk`/`day-for-night` intentionally recolour skin and are documented as best for non-portrait footage. All 24 registry looks are surfaced in the native Theme popup (`fm/cg-native-theme-port`): `CG_THEME_*` in `ColorGrade.h` orders them None(1)/Teal-Orange(2)/Warm-Film(3)/Cool-Noir(4)/Reference Match(5) then the 20-look library at 6-25; `ThemeFromPopup` + the editor's `kThemeNames[]` (EditorWindow.cpp) switch on those named constants and map each to a `getTheme` builder in `Themes.h`. SHARP EDGE (captain decision [key=none-first]): a bare AE popup stores its selection as a 1-based INDEX, so putting None first DELIBERATELY broke the earlier Teal-first stored mapping - there is no safe project-compatible reorder of a keyframeable AE popup (a new None=1 is indistinguishable from an old Teal=1 without a per-project version stamp, and value-rewrite migration would have to rewrite keyframes). Projects saved before this shift have their theme selection permuted and must be re-picked once (accepted as pre-release). Reorder future looks by appending only. The popup default stays Teal (value 2) so a fresh effect keeps the look the Strength/Skin defaults mirror. Adding popup CHOICES needs NO `AE_Effect_Version` bump; reordering does not either (AE keys the param by index, not label). Two cross-cutting findings from that pass: `vlog-indoor-flat-tricky-skin` is an uncorrected-tungsten-WB outlier that drives outsized cast readings on all 3 themes (a fixture/Correct-stage gap, not a theme problem); and `rec709-outdoor` / `vlog-outdoor-bright` show a real ~10-14deg skin hue shift under all 3 themes, from partial-weight pixels at the soft skin-wedge's edge (`skinWedgeWeight` in `src/core/analysis/stats.ts`) getting only partial protection by design - a candidate for the deferred corrective-pinning improvement (see `CONTEXT.md`), not something theme-data tuning alone can close.

- `cep.fs` paths must be plain native OS paths, but `__adobe_cep__.getSystemPath('userData')` returns a `file://` URI on Windows (e.g. `file:///C:/Users/.../AppData/Roaming`). Feeding that URI straight to `cep.fs.writeFile` is what caused the Windows-AE `cep.fs.writeFile failed (err 1)` on the Correct Decode LUT - NOT payload size or a missing encoding argument (a plain path writes a 7MB .cube with no encoding arg just fine; confirmed in the AE CEP console). `cepBridge.ts` `stageTempFile` normalizes via the pure, unit-tested `systemPathToNative` (`src/host/cepPath.ts`, tested in `tests/unit/cepPath.test.ts`): strips the `file://` scheme, `decodeURIComponent`s (`%20`), and drops the leading `/` before a Windows drive letter. Apply this helper to any `getSystemPath`-derived path before it reaches `cep.fs`. Paths from ExtendScript `Folder.temp.fsName` (the render read-back in `cepFrameFileIO.ts`) are already native, so they need no normalization. This boundary is untestable here - the captain verifies in AE via the Correct smoke-checklist item in `docs/cep-install.md`.
- Frame render range (issue: "render produced no file" on timecode-offset comps): `CG_renderFrameToTiff` in `host.jsx` renders one frame via the render queue, and `RenderQueueItem.timeSpanStart` lives in the comp's `displayStartTime` domain, whereas `time` (from `comp.time`) is comp-internal 0-based. On a comp with a large non-zero `displayStartTime`, setting `timeSpanStart = time` (0-based) falls outside the render range -> no frame. Fix: `timeSpanStart = comp.displayStartTime + time`. The PNG fallback's `comp.saveFrameToPng(time, ...)` takes comp-internal time and needs NO shift. Verified via the non-zero-Start-Timecode smoke-checklist item.

- Grade stack (issue #7): the pure bake seam is `lut/gradeLut.ts` (`bakeGradeLut` = `bakeLut(buildTransform(stats, theme, opts))`, default 33-point grid). - Chroma-overshoot guard (`toneStretchChromaGuard` in `engine.ts`, folded into `buildTransform`): stat-matching a low-dynamic-range / degraded source toward a wide-range target used to neon-blow-up chroma - the tone-curve stretch relocates mid pixels into near-empty shadow/highlight bands where auto `bandScale` (<=2x) and the per-pixel `kA`/`kB` std-ratio gain (<=1.8x) compound (scout root cause: `firstmate/data/cg-agent-grade-s7/report.md` Exp A). The guard keys on `stretch = target luma range / source luma range (p5..p95)`: inert below `STRETCH_KNEE=1.5` (ordinary sources -> byte-identical output), ramping to a `gain` floor of 0.2 on the *auto* amplification by `stretch~3`, plus an auto LAB-chroma soft ceiling (3-6x target max band chroma) only when the theme authored no `chromaShape.softLimit`. In the engaged large-stretch regime the guard's `gain` bounds the *total* auto-driven chroma amplification by design - this includes an authored `chromaGain` (it folds into the same `scale` product), since this is exactly the blowout-prevention regime; the auto soft ceiling, though, still defers to an authored `chromaShape.softLimit`, and outside the guard's engaged regime hand-authored intent is untouched. Ported to the native C++ engine in Phase 2 (`toneStretchChromaGuard` in `native/ColorGradeFX/core/Engine.h`, folded into `buildTransform`).

All AEGP wrapped in `catch(...)` (absent gracefully on non-AEGP hosts); requires the SDK's **`AEGP_SuiteHandler.cpp`** compiled in + a client-provided `AEGP_SuiteHandler::MissingSuiteError()` (throws). - Native Phase 4 - live clip preview (`native/ColorGradeFX/editor/PreviewCache.h` + the idle-hook driver in `ColorGrade.cpp` + the D3D texture path in `EditorWindow.cpp`): the editor window shows the actual clip frame, centered + letterboxed, refreshing on timeline scrub and any effect-param change. Design + AE-verification detail is in `native/docs/adr-editor-ui.md` (the "Phase 4" section). Key points: the frame is checked out DOWNSTREAM of this effect (`AEGP_LayerRenderOptionsSuite2::AEGP_NewFromDownstreamOfEffect` -> `AEGP_RenderSuite5::AEGP_RenderAndCheckoutLayerFrame`), so the pixels are already decode+graded - **the V-Log decode invariant holds by construction**, no second decode path. `AEGP_NewFrom*` seeds the options' Time to the layer's current time, read back via `AEGP_GetTime` as the scrub position (no comp-time lookup); `NewFromDownstreamOfEffect` is UI-thread-only, satisfied by the idle hook. The 8-bit ARGB receipt world is copied ARGB->RGBA (alpha forced 255 - premult opaque clip), decimated to <=960px long side, uploaded to a per-window `DXGI_FORMAT_R8G8B8A8_UNORM` texture on the window's UI thread, drawn with `ImDrawList::AddImage` + `letterboxFit`. Interactivity comes from a bounded per-instance LRU cache (16 frames) keyed by (frame time + grade-param fingerprint from the same stream poll that feeds the controls); the pure state machine `decidePreviewAction` picks UpToDate/ServeCached/Render. Checkout is SYNCHRONOUS on the idle thread (sanctioned because the cache makes repeats free); **every** checkout is checked back in unconditionally via a `ScopedCheckin` RAII guard (a leaked receipt destabilizes AE - load-bearing). PORTING/TESTING: the risky logic (cache keying/eviction, scrub/param decision, letterbox fit, once-and-always checkin) is pure in `PreviewCache.h`, headless-tested by `npm run native:preview-test` (`tests/editor/preview_test.cpp`, g++/clang, NOT in CI); the AEGP checkout + D3D upload are captain-verified in AE. All 4 configs (Debug/Release x CPU/`--gpu`) compile clean. DELETION LIFECYCLE (captain findings, rounds 1-2 - CRITICAL): deleting the effect while its window is open does NOT reliably reach `SEQUENCE_SETDOWN`, so the idle hook must self-detect it. NEVER touch ANY stream of a possibly-stale `AEGP_EffectRefH`: a stale ref makes `AEGP_GetNewStreamValue` raise modal 5027::247 (NO_DATA) AND `AEGP_GetNewEffectStreamByIndex`/`AEGP_GetStreamType`-by-index raise "invalid index in indexed group" then CRASH AE. Final design: the idle hook keeps NO AEGP handles at all (effect ref, layer handle, AND comp handle all go stale across delete/undo). At button-open it captures only STABLE IDS: the comp's project-item id (`AEGP_GetItemFromComp`+`AEGP_GetItemID`) + the layer-ID (`AEGP_GetLayerID`) + our installed-effect key. Each tick `ResolveLiveEffect` re-resolves everything from those ids (TRI-STATE, liveness = genuine PROJECT/COMP MEMBERSHIP not call-failure): (1) re-resolve the comp handle by walking the LIVE project item list (`AEGP_GetFirstProjItem`/`AEGP_GetNextProjItem`) for our item id (`ResolveCompByItemID`); (2) `AEGP_GetLayerFromLayerID` within that comp; (3) enumerate that layer's effects by installed key. Alive (use fresh ref for read/write/render, disposed at end of pass; reset counter) / ConfirmedGone (live comp+layer, no effect match => close promptly) / CannotVerify (comp or layer not a live member = comp/layer/source deleted, or partial context). WHY membership not call-failure (rounds 4-5): a deleted layer OR comp zombies in AE's undo buffer where a cached handle keeps resolving, so failed-enumeration counting never fired and the window lingered; re-resolving from the live project/comp tree by id fails on delete and re-succeeds on undo with the CURRENT handle - which also fixes the intermittent edit-after-undo modal by re-deriving the live effect for writes. A single CannotVerify never force-closes (transient guard); a PERSISTENT run closes at `CG_VERIFY_FAIL_LIMIT` (~0.8s).

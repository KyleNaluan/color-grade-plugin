# Project agent memory

This file is the project's committed home for project-intrinsic agent knowledge: build, test, release, architecture, and sharp-edge notes that should travel with the code.

- Domain vocabulary lives in `CONTEXT.md`; use those terms verbatim. Product context: PRD in issue #1 and `docs/prd.md`; ADRs in `docs/adr/`.
- Layout: `src/core` (pure color math), `src/themes` (data-only looks), `src/panel` (Preact UI), `src/host` (typed bridge boundary + CEP impl), `src/extendscript` (ES3 `host.jsx`, AE DOM ops only, zero color math).
- Pixel access goes through the `FrameSource` interface (`src/host/frameSource.ts`, ADR 0001): comp time in, normalized Float32 pixels out - consumers never learn the acquisition method. v1 backend is render-to-file (`renderFrameSource.ts` + `cepFrameFileIO.ts` + ExtendScript `CG_renderFrame`, 16-bit TIFF primary / PNG fallback, decoders in `frameFile.ts`). Tests use the file-backed `fileFrameSource.ts` fed with fixtures, so pixel consumers run without a host.
- Commands: `npm test` (Vitest), `npm run lint` (core-purity check), `npm run typecheck`, `npm run build` (Vite + `scripts/assemble-cep.mjs` -> installable CEP extension in `dist/`; install steps in `docs/cep-install.md`). Theme-tuning loop: `npm run spike -- <frame.tif> <theme>` (see `scripts/spike.ts` header). The spike loop also prints a "grade impact" report (`scripts/lib/gradeImpact.ts`): skin-region hue/chroma shift and overall cast magnitude/direction, applying the built transform to real pixels - use this as the evidence source for theme tuning decisions, not eyeballing stats alone. CI (`.github/workflows/ci.yml`) runs `npm test`, `npm run lint`, and `npm run typecheck` on Node 22 for pushes and PRs to `main`.
- Themes are data only (target stats + authored overrides + knobs); all look logic lives in `src/core/engine/engine.ts`. Strength 0 must stay exact identity and output bounded in [0,1] - tests enforce both. All 3 shipping themes (`src/themes/{teal-orange,warm-film,cool-noir}.ts`) were tuned against the 12 fixture families via the spike loop (issue #6); each theme file's header comment records the before/after evidence and which footage families it suits. Two cross-cutting findings from that pass: `vlog-indoor-flat-tricky-skin` is an uncorrected-tungsten-WB outlier that drives outsized cast readings on all 3 themes (a fixture/Correct-stage gap, not a theme problem); and `rec709-outdoor` / `vlog-outdoor-bright` show a real ~10-14deg skin hue shift under all 3 themes, from partial-weight pixels at the soft skin-wedge's edge (`skinWedgeWeight` in `src/core/analysis/stats.ts`) getting only partial protection by design - a candidate for the deferred corrective-pinning improvement (see `CONTEXT.md`), not something theme-data tuning alone can close.
- `src/core` purity is enforced by `scripts/check-core-purity.ts` (custom checker, not eslint: typescript-eslint's peer range rejects this repo's TypeScript 7). Core may only import relative modules inside `src/core`.
- Golden tests read local-only fixture frames (`tests/fixtures/frames/`, gitignored personal footage) and must keep skipping cleanly when the directory is empty; full suite counts differ with/without fixtures. Local copies live at `~/projects/color-grade-plugin/tests/fixtures/frames/`. Distinct from those: `tests/fixtures/frame-source/` holds small committed synthetic frames (TIFF + PNG) for the FrameSource/analyze tests - regenerate via `scripts/gen-frame-fixtures.ts`.
- The ExtendScript layer is untestable by design; changes there go through the manual smoke checklist in `docs/cep-install.md`.
- Correct stack (issue #5): `src/core/color/decode.ts` (`decodePixelToRec709`) is the single per-pixel decode-to-Rec.709 composition, shared by `analysis/stats.ts` (bulk footage decode) and `lut/decodeLut.ts` (`bakeDecodeLut`, which bakes it into a Decode LUT). Panel orchestration is `panel/correctStack.ts`; the bridge command is `Bridge.setCorrectProfile`, implemented in ExtendScript as `CG_setCorrectProfile` (matchNames `ADBE Apply Color LUT2`, `ADBE Lumetri`). Decode LUTs default to a 65-point grid, not the theme-grade 33-point default: a log profile's highlight rolloff is steep enough that 33 points under-resolves it (see `bakeDecodeLut`'s doc comment and `tests/unit/decodeLut.test.ts`). The Correct tab's UI is a single footage-profile `<select>` (`data-testid="footage-profile"`) that is the one source of truth for the clip's log format: it drives BOTH the analyze decode and, via an explicit **Apply correction** button (`apply-correct`), the Correct stack. `panel/correctStack.ts`'s `applyCorrectForProfile(bridge, profileKey, layerId)` resolves the key to `PROFILES[key]` and decides log-vs-standard via `isLogProfile` (`src/core/color/index.ts`, standard = `STANDARD_PROFILE_KEY='rec709'`) - no hardcoded profile in the apply path; a log key bakes+applies that profile's Decode LUT, `rec709` removes it. The selector resets to `rec709` per-clip and is guarded by `applyGeneration`/`targetKey` against stale applies. Extending to more camera profiles (`multi-camera-profiles`) is just adding entries to `PROFILES`.
- `cep.fs` paths must be plain native OS paths, but `__adobe_cep__.getSystemPath('userData')` returns a `file://` URI on Windows (e.g. `file:///C:/Users/.../AppData/Roaming`). Feeding that URI straight to `cep.fs.writeFile` is what caused the Windows-AE `cep.fs.writeFile failed (err 1)` on the Correct Decode LUT - NOT payload size or a missing encoding argument (a plain path writes a 7MB .cube with no encoding arg just fine; confirmed in the AE CEP console). `cepBridge.ts` `stageTempFile` normalizes via the pure, unit-tested `systemPathToNative` (`src/host/cepPath.ts`, tested in `tests/unit/cepPath.test.ts`): strips the `file://` scheme, `decodeURIComponent`s (`%20`), and drops the leading `/` before a Windows drive letter. Apply this helper to any `getSystemPath`-derived path before it reaches `cep.fs`. Paths from ExtendScript `Folder.temp.fsName` (the render read-back in `cepFrameFileIO.ts`) are already native, so they need no normalization. This boundary is untestable here - the captain verifies in AE via the Correct smoke-checklist item in `docs/cep-install.md`.
- Frame render range (issue: "render produced no file" on timecode-offset comps): `CG_renderFrameToTiff` in `host.jsx` renders one frame via the render queue, and `RenderQueueItem.timeSpanStart` lives in the comp's `displayStartTime` domain, whereas `time` (from `comp.time`) is comp-internal 0-based. On a comp with a large non-zero `displayStartTime`, setting `timeSpanStart = time` (0-based) falls outside the render range -> no frame. Fix: `timeSpanStart = comp.displayStartTime + time`. The PNG fallback's `comp.saveFrameToPng(time, ...)` takes comp-internal time and needs NO shift. Verified via the non-zero-Start-Timecode smoke-checklist item.
- Apply Color LUT's file-picker property is not officially documented as scriptable; both `CG_ensureDecodeLut` (Correct) and `CG_ensureGradeLut` (Grade) set it via `effect.property(1).setValue(file)` as the best-known approach - verify it actually resolves the LUT file each release via the smoke checklist.
- Grade stack (issue #7): the pure bake seam is `lut/gradeLut.ts` (`bakeGradeLut` = `bakeLut(buildTransform(stats, theme, opts))`, default 33-point grid). Panel orchestration is `panel/gradeStack.ts` (`applyThemeGrade`); the bridge command is `Bridge.applyGrade`, implemented as `CG_applyGrade`, which writes `grade_<layerId>.cube` + its recipe `grade_<layerId>.json` into `.colorgrade/` and puts one Managed `[cg]` adjustment layer at the top of the comp carrying an Apply Color LUT effect. Grade analysis is post-Correct by construction: the FrameSource renders the comp *with* the clip's Correct stack applied, so `applyThemeGrade` decodes through the identity `REC709` profile (no second software decode) - this is the hard pipeline-order requirement. On a re-grade, `applyThemeGrade` disables the existing `[cg]` grade layer (via `Bridge.setGradeLayerEnabled` / `CG_setGradeLayerEnabled`) around the analysis render and re-enables it in a `finally`, so the analyzed pixels stay post-Correct instead of feeding the prior grade back in. Knob recompute-from-cached-stats and per-clip grade recall are deferred (issues #11+); the tracer bakes with default knobs and persists the recipe inputs for that later work.
- Chroma-overshoot guard (`toneStretchChromaGuard` in `engine.ts`, folded into `buildTransform`): stat-matching a low-dynamic-range / degraded source toward a wide-range target used to neon-blow-up chroma - the tone-curve stretch relocates mid pixels into near-empty shadow/highlight bands where auto `bandScale` (<=2x) and the per-pixel `kA`/`kB` std-ratio gain (<=1.8x) compound (scout root cause: `firstmate/data/cg-agent-grade-s7/report.md` Exp A). The guard keys on `stretch = target luma range / source luma range (p5..p95)`: inert below `STRETCH_KNEE=1.5` (ordinary sources -> byte-identical output), ramping to a `gain` floor of 0.2 on the *auto* amplification by `stretch~3`, plus an auto LAB-chroma soft ceiling (3-6x target max band chroma) only when the theme authored no `chromaShape.softLimit`. In the engaged large-stretch regime the guard's `gain` bounds the *total* auto-driven chroma amplification by design - this includes an authored `chromaGain` (it folds into the same `scale` product), since this is exactly the blowout-prevention regime; the auto soft ceiling, though, still defers to an authored `chromaShape.softLimit`, and outside the guard's engaged regime hand-authored intent is untouched. Ported to the native C++ engine in Phase 2 (`toneStretchChromaGuard` in `native/ColorGradeFX/core/Engine.h`, folded into `buildTransform`).
- Native re-platform (`native/`, Windows-first; direction in `firstmate:native-scope-m2/report.md`): Phase 1 ships `ColorGradeFX.aex`, a SmartFX effect applying a baked 3D LUT via a C++ port of `sampleLut`. Build/toolchain/env-vars are in `native/BUILDING.md` (the committed source of truth, superseding firstmate's `native-phase0-*.md`); layout in `native/README.md`. Build from WSL with `native/scripts/build.sh [Debug|Release] [--gpu]` - it mirrors `native/` to NTFS (`C:\dev\cg-native-build`) and drives VS 2022 `MSBuild.exe` over interop; the `.aex` lands in `AE_PLUGIN_BUILD_DIR` (MediaCore). Close AE before rebuilding. `native/ColorGradeFX/lut/CubeLut.h` is the ported `parseCube`+`sampleLut` and MUST stay numerically identical to `src/core/lut/cube.ts` (the oracle); `npm run native:parity` gates them to ~1e-4 (needs g++/clang, deliberately NOT in CI). `embedded/EmbeddedLut.h` is generated by `npm run native:gen-lut` (a real teal-orange grade, 17^3, a float array not a string to dodge MSVC's 64KB literal cap) and is committed. One vcxproj serves CPU (default) and GPU (`/p:CG_GPU=true`, which builds BOTH DirectX and CUDA); the GPU kernel is `ColorGrade_Kernel.cu` (GF macros): DXC compiles it for DirectX (LUT as a second raw-SRV t1), nvcc compiles it for CUDA (LUT via cudaMalloc/cudaMemcpy). DirectX flags must match in BOTH `GlobalSetup` out_flags2 AND the PiPL (`0x2a001400`, flipped via `#ifdef HAS_HLSL`), and `DirectX_Assets/` must ship next to the `.aex`, or AE silently uses CPU. IMPORTANT runtime finding: AE 2025 on the dev box (RTX 3090 + AMD) offers only Mercury CUDA/OpenCL/Software - NO DirectX - so CUDA is the path that actually engages there; DirectX is kept for other hosts, OpenCL deferred. CUDA link ignores libcmt/libcmtd (cudart_static's static CRT) so the dynamic CRT wins. Charset is MultiByte (SDK `DirectXUtils.cpp` needs it). Which framework AE offers is logged at `GPUDeviceSetup` (Debug tracer, see `native/BUILDING.md` "Verifying the GPU path"). AE runtime behavior is captain-verified only.
- Native Phase 2 (`native/ColorGradeFX/core/*.h`): the whole `src/core` engine ported to C++ (header-only, dependency-free, `cg::core` namespace) - color/lab/vlog/rec709, `MonotoneCurve` (PCHIP), `computeStats`, `buildTransform`, `bakeLut`/`bakeGradeLut`/`bakeDecodeLut`, plus `Themes.h` (the 3 shipping themes transcribed as data) and `Recipe.h` (the arb-data POD grade recipe + `<->` Theme/stats + `bakeFromRecipe`). PORTING RULE: all engine math is `double` (JS `number`); narrow to float ONLY where the TS does - the `cg::Lut3D` grid and `computeStats`' luma `Float32Array`. `src/core` stays the oracle; port faithfully, never "improve" numerics. Gate: `npm run native:core-parity` (`native/scripts/core-parity-test.ts` + `tests/parity/core_parity.cpp`) replays computeStats/bakeGradeLut/bakeDecodeLut/recipe in both engines - achieved grade/decode/recipe **bit-exact**, stats ~2e-13 (tol 1e-4); g++/clang, NOT in CI. The effect (`ColorGrade.cpp`) gained a Phase 2 param surface (Theme popup, Strength/Skin-Protection/Chroma-Gain sliders, LUT-Source Auto|Embedded|External) + a **PF_Cmd_ARBITRARY_CALLBACK** arb-data handler persisting the POD `RecipeData` (all memcpy/memcmp - it's pointerless POD; refcon+magic+version guarded; pattern from SDK `ColorGrid_Arb_Handler.cpp`). "Auto" bakes the grade in-effect from theme+recipe. SHARP EDGES: (a) the core headers use `std::min/std::max`/`std::optional`, so the vcxproj MUST define **`NOMINMAX`** (added) - else `<Windows.h>`'s min/max macros wreck every `std::max` call and cascade into `<optional>`; LanguageStandard is already `stdcpp17`. (b) `ColorGradePiPL.r`'s `AE_Effect_Version` MUST track `PF_VERSION(MAJOR,MINOR,...)` in `ColorGrade.h` (currently 1.1 -> `557057`) or AE warns "version mismatch (88001)". (c) the data-only `Grade Recipe` arb param MUST be `ui_width=0,ui_height=0` + `PF_PUI_NO_ECW_UI`, else AE warns 25::37 + "Unsupported effect control!". (d) build ALL FOUR configs pre-validation (Debug/Release x CPU/`--gpu`) - a config-gated `ERR2`/`err2` decl break compiles in one config and fails another (a Phase-1 no-mistakes auto-fix left `err2` undeclared in PreRender on main; fixed here); back-to-back builds can hit transient LNK1201 (stale PDB) - clear `mirror/.../x64/<Config>` + `mspdbsrv.exe`. See `native/BUILDING.md`. CPU `.aex` builds clean via `build.sh Debug`; GPU build + AE runtime are captain-verified. `Engine.h`'s `buildTransform` tracks main including the auto chroma-overshoot guard `toneStretchChromaGuard` (PR #23) and the `midtoneTint` `ThemeOverrides` field (PR #24) - both ported, with harness coverage that actually exercises them (a low-dynamic-range frame drives the guard to full severity; `grademid` injects a midtone tint since no shipping theme sets one). PORT-TRACKING: when a `src/core/engine` or `theme` change lands on main, re-port it into `Engine.h`/`Theme.h`/`Recipe.h` and re-run `native:core-parity`; if the port adds/reorders `RecipeData` fields, bump `RECIPE_VERSION` (currently 2) so the arb-data unflatten guard reseeds stale blobs.
- Native Phase 3 - editor window (`native/ColorGradeFX/editor/` + `native/third_party/imgui/`): a native editor window opened from an **Open Editor…** button param (`CG_OPEN_EDITOR`, SUPERVISE), hosting the full **Correct/Grade control set**. **Toolkit decision (captain-approved) is `native/docs/adr-editor-ui.md`**: **Dear ImGui + Win32/D3D11** (vendored v1.91.5, compiled into the `.aex`, zero runtime dep, single-file drop-in) over an embedded webview reusing Preact - webview loses on Windows packaging (WebView2 runtime) + the Phase 4-5 live-GPU-preview/scopes latency, which are the axes that matter. Controls: Correct tab = **Footage** popup (`CG_FOOTAGE_PROFILE`: Rec.709 / V-Log); Grade tab = Theme / Strength / Skin Protection / Chroma Gain / LUT Source. The Footage param drives a native **decode stage** applied in EVERY LUT Source mode (pipeline order always decode-then-LUT, captain directive): Auto composes `grade(decodePixelToRec709(x))` into one LUT (continuous), Embedded/External resample their raw LUT through decode (`ComposeDecodeIntoLut`: `sampleLut(rawLut, decode(x))`); both keep the per-pixel apply a single trilinear sample so CPU/GPU stay identical (Rec.709 = no-op). Proven vs the TS oracle by the `gradedecode` (bit-exact) and `lutdecode` (3.3e-7) cases in `npm run native:core-parity`. Architecture: `EditorBridge.h` is the pure, AE/Win32-free seam (thread-safe coalescing edit queue + `EditField` + `ParamSnapshot` + percent<->fraction mapping + `applyEdit`), headless-tested by `npm run native:editor-test` (`tests/editor/bridge_test.cpp`, g++/clang, NOT in CI). `EditorWindow.{h,cpp}` is the Win32/D3D11/ImGui host: single-instance-per-effect registry (keyed on `in_data->effect_ref`; production wants a sequence-data uid), each window on its own UI thread; `.cpp` is no-op stubs off Windows for the eventual Cocoa/Metal Mac backend. INSTANCE KEY (`SeqKey`): the registry is keyed on a uid in **sequence data** (flat POD `CG_SequenceData`, reseeded on SEQUENCE_SETUP/RESETUP so duplicated/reloaded effects never share a key). This fixed a live-AE bug: `in_data->effect_ref` is NOT stable across command types (differs USER_CHANGED_PARAM vs SMART_PRE_RENDER), so effect_ref-as-key made publish/drain miss the window and the round-trip was dead. Bridge flow: effect->window `publishSnapshot` from BOTH `PreRender` (opportunistic, locked value copy) AND the idle hook, which POLLS the effect's param streams (`AEGP_GetNewStreamValue`) and publishes on change - the idle-hook poll is the reliable EC->window path because `PreRender`'s `sequence_data`/key is unreliable on render threads (this was the last EC->window-not-updating fail); monotonic `revision` + the window's `IsAnyItemActive` mid-drag guard prevent stomping an active drag. window->effect edits applied by the same **global AEGP idle hook** registered via PICA (`AEGP_RegisterSuite5::AEGP_RegisterIdleHook` after `AEGP_RegisterWithAEGP` - NO separate AEGP plugin/PiPL) that drains each open window and writes streams (`AEGP_StreamSuite5` GetNewEffectStreamByIndex+SetStreamValue) inside one `AEGP_StartUndoGroup`/`EndUndoGroup` per tick; it reaches the instance via a per-key `AEGP_EffectRefH` captured on button-open (`AEGP_PFInterfaceSuite1::AEGP_GetNewEffectForEffect`) and cheap-noops when nothing is pending. All AEGP wrapped in `catch(...)` (absent gracefully on non-AEGP hosts); requires the SDK's **`AEGP_SuiteHandler.cpp`** compiled in + a client-provided `AEGP_SuiteHandler::MissingSuiteError()` (throws). The in-`UserChangedParam` `CHANGED_VALUE` drain stays as a secondary flush. Lifecycle: `GLOBAL_SETDOWN`->`shutdownAll`, `SEQUENCE_SETDOWN`->close+dispose effect ref (no orphan windows). VCXPROJ: links `d3d11.lib;dxgi.lib` (imgui backends `#pragma comment(lib)` d3dcompiler/gdi32/dwmapi; XInput dynamic), compiles `AEGP_SuiteHandler.cpp`, adds `third_party/imgui[/backends]` includes; editor sources compile in ALL FOUR configs (build Debug/Release x CPU/`--gpu`); NOMINMAX/MultiByte/stdcpp17 already satisfy ImGui. LINK-VERIFY WITHOUT CLOSING AE: pass `-p:AE_PLUGIN_BUILD_DIR=C:\dev\cg-verify-out` (a WSL `export` does NOT cross to Windows MSBuild - must be an explicit `-p:` prop) to relink to a scratch dir instead of the locked MediaCore `.aex`. DEFERRED (captain-verified in AE, later phases): live preview (Phase 4, now landed - see next bullet) + scopes (Phase 5). AEGP DOM round-trip + undo are captain-verified in AE. Adding a param shifts the `CG_*` enum but all refs are by name; PiPL `AE_Effect_Version` unchanged, so no version-mismatch risk.
- Native Phase 4 - live clip preview (`native/ColorGradeFX/editor/PreviewCache.h` + the idle-hook driver in `ColorGrade.cpp` + the D3D texture path in `EditorWindow.cpp`): the editor window shows the actual clip frame, centered + letterboxed, refreshing on timeline scrub and any effect-param change. Design + AE-verification detail is in `native/docs/adr-editor-ui.md` (the "Phase 4" section). Key points: the frame is checked out DOWNSTREAM of this effect (`AEGP_LayerRenderOptionsSuite2::AEGP_NewFromDownstreamOfEffect` -> `AEGP_RenderSuite5::AEGP_RenderAndCheckoutLayerFrame`), so the pixels are already decode+graded - **the V-Log decode invariant holds by construction**, no second decode path. `AEGP_NewFrom*` seeds the options' Time to the layer's current time, read back via `AEGP_GetTime` as the scrub position (no comp-time lookup); `NewFromDownstreamOfEffect` is UI-thread-only, satisfied by the idle hook. The 8-bit ARGB receipt world is copied ARGB->RGBA (alpha forced 255 - premult opaque clip), decimated to <=960px long side, uploaded to a per-window `DXGI_FORMAT_R8G8B8A8_UNORM` texture on the window's UI thread, drawn with `ImDrawList::AddImage` + `letterboxFit`. Interactivity comes from a bounded per-instance LRU cache (16 frames) keyed by (frame time + grade-param fingerprint from the same stream poll that feeds the controls); the pure state machine `decidePreviewAction` picks UpToDate/ServeCached/Render. Checkout is SYNCHRONOUS on the idle thread (sanctioned because the cache makes repeats free); **every** checkout is checked back in unconditionally via a `ScopedCheckin` RAII guard (a leaked receipt destabilizes AE - load-bearing). PORTING/TESTING: the risky logic (cache keying/eviction, scrub/param decision, letterbox fit, once-and-always checkin) is pure in `PreviewCache.h`, headless-tested by `npm run native:preview-test` (`tests/editor/preview_test.cpp`, g++/clang, NOT in CI); the AEGP checkout + D3D upload are captain-verified in AE. All 4 configs (Debug/Release x CPU/`--gpu`) compile clean. DELETION LIFECYCLE (round-1 captain finding): deleting the effect while its window is open does NOT reliably reach `SEQUENCE_SETDOWN`, so the idle hook probes `EffectRefAlive` (an `AEGP_GetStreamType` on CG_THEME - safe on a stale ref, unlike `AEGP_GetNewStreamValue` which raises AE modal 5027::247 on the resulting `NO_DATA` stream) at the top of each per-window pass and closes the window + disposes the ref on a `NO_DATA` type; `ReadStreamOneD` also rejects `NO_DATA` defensively. DEFERRED: async checkout (`AEGP_RenderAndCheckoutLayerFrame_Async` - SDK prefers it for passive redraws; the state machine is already shaped for one in-flight request) if scrub interactivity needs it; the cache fingerprint keys on time + OUR params, so an unrelated edit (another effect / source swap without a time change) can leave the preview stale until the next scrub/param nudge; scopes + before/after are Phase 5.
